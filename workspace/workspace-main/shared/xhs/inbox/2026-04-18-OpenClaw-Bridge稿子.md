## 产品背景

**Bridge** 是翔宇工作流系列第 17 期的核心文章主题，解决的是 OpenClaw Agent 如何稳定调用 Claude Code 的问题。

核心痛点：OpenClaw 的 10 个 AI Agent 各就各位（情报、研发、运维、小红书、微信、公众号、视频、课程、行政），但 Agent 能调用 Claude Code ≠ 能可靠地调用。进程超时、崩溃、参数不全、无人应答——这些都是真实踩过的坑。

Bridge 就是解决这个问题的「桥」。

## 期望交付
一篇小红书笔记，介绍 Bridge 这个工具/系统。

## 内容要点（从原文提炼）

1. **问题切入**：Agent 调用 Claude Code，听起来很美，但实际跑起来——进程超时没人管、崩溃了 Agent 不知道、凌晨 3 点卡在"请输入……"上等一整夜。**「能调用不等于能可靠调用」**

2. **Bridge 是什么**：Agent（老板）和 Claude Code（工程师）之间的秘书。接到老板指令，找到工程师，盯着干完，把结果带回来汇报。

3. **三层架构**：
   - 第一层（bridge-runner.sh）：调度室，接单、登记、安排
   - 第二层（bridge.sh）：门卫，检查环境是否就绪
   - 第三层（bridge.py）：执行间，用 Claude Agent SDK 启动 CC 进程

4. **两种模式**：
   - 同步=等快递（阻塞等待）
   - 异步=寄快递（派单后立即返回，任务后台跑，Agent 继续处理其他消息）
   - 核心是「不阻塞」，10 个 Agent 并行干活

5. **三层保险**：
   - 自动善后：任务怎么结束都记录
   - 万能兜底：崩溃也捕获
   - 看门狗：超时强制终止
   - 任务永远不会「失踪」

6. **半夜没人回答怎么办**：所有答案提前打包好，CC 需要什么自己去找，不会卡在交互提问上

7. **干完了自动汇报**：结果自动送到 Agent，Agent 在 Discord 频道回复。老板端着咖啡等结果就行

8. **金句**：搭建 Agent 是 10%，让 Agent 可靠地干活是 90%。Bridge 是那 90% 最关键的一块拼图
